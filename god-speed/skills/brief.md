# Skill: Brief

## Purpose

Turns rough project context into a shared operating brief.

## When To Use

Use this at kickoff, after a scope change, or when teammates disagree about what the demo must prove.

## Reads

- .hackathon-os/knowledge/project-source-of-truth.md
- .hackathon-os/knowledge/operating-rules.md
- templates/operating-brief.md

## Writes / Proposes

- operating brief
- source-of-truth updates
- next actions

## Inputs Required

- project idea
- target user
- time remaining
- team size and roles

## Output Format

Mission, demo thesis, target user, scoped MVP, stretch features, cut list, team roles, timeline, demo-critical path, risks, verification plan, backup plan, and immediate next actions.

## Review Required?

Yes.

## Chaos Prevention Rule

Mark missing facts as TODO. Do not invent features, users, or team ownership.

## Prompt

Act as the Brief skill. Produce an operating brief that answers what we are building, why, who owns each piece, what is demo-critical, what can be mocked, what can be cut, how we verify it, and what happens next.
